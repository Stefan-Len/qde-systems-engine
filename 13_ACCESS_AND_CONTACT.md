# Access and Contact

If, after the previous sections, QDE is interesting to you, the next step is not
that you immediately receive a public download link.

This is not a mass-market product and not a public bot download.

I want to handle access to the QDE commercial source-code package in a
controlled way: first form, buyer context, personal review, terms, order flow,
and only then delivery if it is clear from all sides what you receive and what
you do not receive.

That is better for you, and better for me.

I do not want to deliver it to someone looking for a signal room, a quick profit
promise, broker helpdesk, or an "install and make money" bot. QDE may be
interesting for people who think in terms of source code, audit, validation,
risk boundary, broker boundary, and agent review.

## What Can An Accepted Buyer Receive?

The precise delivered scope is always determined by the current purchase terms,
package scope, delivery manifest, and checksum.

At a high level, the QDE commercial package direction is:

- source-code-based deterministic execution infrastructure;
- Python backend and macOS operator UI source;
- risk gate, audit trail, execution boundary, and mode workflow;
- ML/model integration and validation-oriented runtime thinking;
- a system surrounded by tests and validation workflows;
- detailed technical and operator documentation;
- helper scripts for setup, build, test, and validation direction;
- broker-adapter-ready architecture, without a production broker adapter;
- agent-review workflow and rule discipline for source-code, docs, audit, and
  change-review work;
- delivery-manifest / checksum-style delivery evidence;
- community / Discord technical peer layer, if the given order or launch
  expressly includes it.

This does not mean every future integration, broker adapter, deployment, custom
strategy, market-data subscription, account setup, or support work is included.

The package is a source-code and infrastructure foundation. It is not a managed
service.

## How To Request The Next Step

If you are interested in QDE access, the first step is to fill out the form.

I personally review every request. This is not automatic checkout and not an
immediate order process.

If the context is aligned, I will contact you and then explain the next step:
which terms need to be reviewed, what order / acceptance flow comes next, how
payment may work, and only after that how delivery may happen.

At this stage, I may also send a short pre-order technical overview or technical
summary. Its purpose is to help you better understand the main parts and
boundaries of the system before moving toward order / acceptance / payment. It
is not source-code delivery, not a license grant, not full private
documentation, and not a support service.

The point of the access request is not a long sales message, but a clean first
context.

It is enough to describe briefly and concretely:

- who you are or which entity you are asking on behalf of;
- your technical / trading / execution background at a high level;
- why QDE is interesting to you;
- whether B2B or consumer buyer status is likely;
- what environment you are thinking in: research, private infrastructure,
  validation, broker adapter work, documentation / audit workflow, agent
  workflow;
- whether you have an environment suitable for a Mac / Xcode / Python local
  workflow;
- whether you understand this is not a signal service, not financial advice, not
  managed trading, and not a ready-to-trade LIVE broker bot.
